# Development Workflow

Disciplined development practices for any coding work. Route to the right sub-skill:

| Skill | When to use |
|-------|-------------|
| **brainstorming** | Before ANY creative work — new features, components, behavior changes. Explore intent before implementation. |
| **writing-plans** | After brainstorming produces a spec. Break it into bite-sized implementation tasks. |
| **test-driven-development** | Before writing implementation code. Tests first, always. |
| **systematic-debugging** | When hitting any bug, test failure, or unexpected behavior. Investigate before fixing. |
| **verification-before-completion** | Before claiming work is done. Run it, prove it, then say it. |
| **requesting-code-review** | After implementation. Self-review using the checklist before presenting work. |
| **receiving-code-review** | When receiving feedback. Verify before implementing. Push back when wrong. |

## Flow

Typical order: brainstorming → writing-plans → test-driven-development → verification → requesting-code-review

Systematic-debugging and receiving-code-review are reactive — use when triggered by failures or feedback.
